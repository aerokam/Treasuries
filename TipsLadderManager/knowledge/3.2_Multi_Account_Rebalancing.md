# 3.2 Multi-Account Rebalancing (Status: Implemented)

## Dependencies

**Requires:** 3.0 TIPS Ladder Rebalancing, 2.1 Broker Import

**Inherits:** All concepts and algorithms from 3.0 (DARA, gap years, bracket years, Phase 1-4, LMI, etc.)

---

## Purpose

Enable rebalancing of TIPS ladders held across **multiple accounts for a single owner**, optimally distributing maturities across accounts to minimize tax drag while respecting account-level cash flow constraints.

**Key constraint:** Proceeds from selling TIPS in one account can only fund purchases in that same account. No inter-account cash transfers.

**Example use case:** An investor holds a TIPS ladder split across a taxable brokerage account ($50k), a Traditional IRA ($200k), and a Roth IRA ($100k). When rebalancing, short-maturity TIPS should preferentially be held in the taxable account (minimize years of reinvested-income tax drag), while long-maturity TIPS should preferentially be held in Traditional IRAs (avoid tax drag entirely). The algorithm optimally allocates the target ladder to these accounts.

---

## Conceptual Approach: Liquidate-and-Rebuild Optimally

Rather than adjusting current holdings in place, multi-account rebalancing answers: **"If we liquidated all TIPS and rebuilt the ladder from scratch, how would we optimally distribute maturities across accounts?"**

### Two-Layer Process

**Layer 1: Blended Rebalance (Unchanged)**
- Merge all holdings across all accounts into a single portfolio
- Run the standard Phase 1-4 algorithm from 3.0 TIPS Ladder Rebalancing
- Output: target quantities per CUSIP per year, as if there were no account constraints

**Layer 2: Optimal Account Allocation (New)**
- Given target quantities from Layer 1 and available account sizes, allocate maturities across accounts
- Maximize tax preference satisfaction: short TIPS in taxable, medium in Roth, long in Traditional
- Respect account capacity constraints (each account has a size limit in dollars, which translates to ~TIPS capacity accounting for index ratios)
- Compute per-account cash flows and flag infeasibility if any account can't balance

---

## Tax Drag Minimization (Core Principle)

**TIPS reinvested income**: Each year, the principal of a TIPS bond adjusts upward by the inflation rate. This adjustment is **real income**, automatically reinvested in the bond. In taxable accounts, you pay income tax on this adjustment even though you receive no cash, creating a **tax drag** that compounds over the holding period.

**Tax preference by maturity:**
- **Short TIPS (first third of ladder)** → **Taxable account preferred**: Reinvested-income tax drag is limited to a few years. Acceptable.
- **Medium TIPS (middle third of ladder)** → **Roth IRA preferred**: Tax-free growth avoids drag entirely.
- **Long TIPS (final third of ladder)** → **Traditional IRA strongly preferred**: Avoids decades of reinvested-income tax drag.

**Rationale**: Shifting long-maturity TIPS to IRAs eliminates the largest present-value tax drag (years × rate).

## Algorithm Flow

### Phase 1-4 (Layer 1): Standard Blended Rebalance

1. Merge all holdings by CUSIP (sum quantities across accounts)
2. Run Phase 1-4 from 3.0 TIPS Ladder Rebalancing on merged portfolio
3. Output: `qtyAfter_blended[cusip][year]` for each CUSIP and funded year

### Phase 5 (Layer 2): Optimal Account Allocation

**Input**: Layer 1 output + holdings with account metadata + account sizes.

#### Step 5.1 — Determine Maturity Tiers

Divide the ladder into three tiers based on maturity position:

```
sortedYears = SORT(all funded years in target ladder)
thirdSize = LENGTH(sortedYears) / 3

shortTier = sortedYears[0 : thirdSize]                    // first third
mediumTier = sortedYears[thirdSize : 2×thirdSize]         // middle third
longTier = sortedYears[2×thirdSize : end]                 // final third
```

#### Step 5.2 — Allocate Respecting Dollar Budgets

For each account, the allocation constraint is **dollars**, not TIPS count:

```
for each account:
  remainingBudget = account.sizeInDollars - totalDollarsAllocatedSoFar
  // Allocate next TIPS up to this budget using per-CUSIP cost
```

This accounts for the fact that longer-maturity TIPS (allocated to IRAs first) have higher index-adjusted costs than shorter-maturity TIPS.

#### Step 5.3 — Optimal Allocation by Tax Preference (Greedy Fill)

Allocate target quantities to accounts using dollar-budget constraints and preference order:

1. **Long tier (Tier 3) → Traditional IRA**: Longest maturities preferentially to Traditional IRA (tax drag avoided for decades)
2. **Medium tier (Tier 2) → Roth IRA**: Medium maturities to Roth (tax-free growth)
3. **Short tier (Tier 1) → Taxable**: Short maturities to taxable (limited tax drag, only few years)

**Algorithm**: Sort all target TIPS by maturity (longest first), then allocate greedily to accounts in preference order:

```
allTargets = SORT(all (cusip, year, qty) by year DESC)
accountBudgetUsed = {}

for each target in allTargets:
  remaining = target.qty
  for each accountType in preference_order:
    for each account of accountType:
      availableBudget = account.sizeInDollars - accountBudgetUsed[account]
      maxAffordable = FLOOR(availableBudget / costPerBond[cusip])
      toAllocate = MIN(remaining, maxAffordable)
      if toAllocate > 0:
        allocation[account][cusip][year] += toAllocate
        accountBudgetUsed[account] += toAllocate * costPerBond[cusip]
        remaining -= toAllocate

if remaining > 0:
  flag_infeasibility(cusip, year, remaining)
```

#### Step 5.4 — Compute Per-Account Current Holdings

For each (account, CUSIP, year), retrieve current holdings from import:

```
qtyBefore[account][cusip][year] = holdings data
```

#### Step 5.5 — Compute Per-Account Feasibility

Under the **liquidate-and-rebuild model**, feasibility is determined by comparing total allocation cost to account budget:

```
for each account:
  // Informational: current TIPS value in this account
  currentValue = SUM(qty × costPerBond[cusip] for all (cusip, year) in currentHoldings[account])
  
  // Feasibility: does new allocation fit within budget?
  newAllocationCost = SUM(qty × costPerBond[cusip] for all (cusip, year) in allocation[account])
  
  shortfall = MAX(0, newAllocationCost - account.sizeInDollars)
  feasible[account] = (newAllocationCost <= account.sizeInDollars)
```

#### Step 5.6 — Global Feasibility Check

```
globalFeasible = ALL(feasible[account] for all accounts)

if NOT globalFeasible:
  infeasibleAccounts = [account | feasible[account] = FALSE]
  // Report shortfall amount for each infeasible account
```

#### Step 5.7 — Output Generation

Generate one output row per (account, CUSIP, year) containing:
- Account name, CUSIP, maturity year
- `qtyBefore`, `qtyAfter`, `qtyDelta` (per-account)
- Sell proceeds or buy needs (derived from `qtyDelta`)
- Feasibility summary (global + per-account)

## Named Quantities & Formulas

### Maturity Tier Assignment

```
sortedYears = SORT(all funded years in [firstYear, lastYear])
tierSize = LENGTH(sortedYears) / 3

year ∈ shortTier   if year ≤ sortedYears[tierSize]
year ∈ mediumTier  if sortedYears[tierSize] < year ≤ sortedYears[2×tierSize]
year ∈ longTier    if year > sortedYears[2×tierSize]
```

### Per-Account Allocation Budget Constraint

```
remainingBudget[account] = account.sizeInDollars - dollarsAllocatedSoFar[account]
maxAffordable[account][cusip] = FLOOR(remainingBudget[account] / costPerBond[cusip])
```

### Quantity After (per-account, optimal allocation)

```
qtyAfter[account][cusip][year] = result of Step 5.3 greedy allocation algorithm
```

### Total Allocation Cost (per-account)

```
newAllocationCost[account] = SUM(qtyAfter[account][cusip][year] × costPerBond[cusip]
                                  for all (cusip, year))
```

### Feasibility

```
feasible[account] = (newAllocationCost[account] <= account.sizeInDollars)
shortfall[account] = MAX(0, newAllocationCost[account] - account.sizeInDollars)
globalFeasible = ALL(feasible[account] for all accounts)
```

## UI Behavior Notes

### File Import
- When a new holdings file is imported, all form fields are cleared to force re-inference from the new holdings:
  - DARA field is unconditionally cleared (not just when auto-inferred)
  - Last Year and First Year fields are cleared
  - Account selection modal is reset
- This prevents stale parameters (especially DARA) from the previous file run from contaminating the new rebalance

### Multi-Account Checkbox
- Auto-checked when multiple accounts are detected in the import
- Auto-unchecked when a single account (or zero accounts) are present
- User can manually toggle, but automatic detection is the default

### Account Size Sliders
- Default value: TIPS value (sum of current value for TIPS holdings in that account)
- Maximum value: Total account value (sum of all holdings, not just TIPS)
- User adjusts with slider or direct input

## Limitations & Edge Cases (v1.0)

1. **No inter-account cash transfers**: If Account A needs to fund more TIPS than its budget, but Account B has excess, rebalance is infeasible. Future: allow user-directed inter-account transfers.

2. **Account type auto-detected**: System detects account type from account name keywords ("ROTH" → Roth IRA, "IRA" → Traditional IRA, else taxable). Manual override not yet available.

3. **Per-CUSIP cost used for allocation**: Allocation respects exact per-CUSIP cost, accounting for index-ratio markup. No average-cost simplification. High-cost (long-maturity) TIPS may reduce affordable quantity.

4. **Tax preference is strong, not absolute**: When an account reaches budget limit, allocation cascades to lower-preference accounts. This ensures feasibility within total available space.

5. **Tier boundaries are relative, not absolute**: "Short," "medium," "long" are defined by maturity position in the ladder, not fixed years. Different ladders have different boundaries.

6. **No partial rebalances**: If overall rebalance is infeasible, the entire result is flagged. Future: allow user to approve partial rebalances per account.

7. **Unified table in multi-account mode**: The main rebalance table (Layer 1 blended) is shown in both single- and multi-account modes. In multi-account mode, account sub-rows are injected under each CUSIP/Year row showing per-account Before / After / BUY|SELL|HOLD. The separate lower allocation table is not used.

## Future Enhancements (Stretch Goals)

1. **Inter-Account Cash Transfers**: Allow user to approve temporary cash flows between accounts (e.g., taxable → IRA contribution).
2. **Per-Account Partial Rebalances**: User can choose to rebalance some accounts and defer others.
3. **Per-Account DARA Targets**: Different DARA per account (e.g., Traditional IRA targets $50k/yr, Roth targets $30k/yr).
4. **Account Precedence Weighting**: User assigns priority (e.g., "rebalance IRA first, taxable is secondary").
5. **Multiple Taxable Accounts**: Distinguish between different taxable accounts (e.g., long-term vs. short-term gains management).

## Implementation Notes

- **Layer 1 reuse**: `runMultiAccountRebalance()` calls `runRebalance()` on merged holdings (all accounts combined). No changes to Phase 1-4.
- **Layer 2 implementation**: `allocateToAccounts()` in `src/account-allocation.js` implements Steps 5.1–5.7.
- **Return value**: `runMultiAccountRebalance()` returns `{ rebalanceResult, accountAllocation, accountMetadata, maturityTiers, accountCashFlows, feasibility, allocationInfeasibilities, currentHoldingsByAccount }`. The `currentHoldingsByAccount` field (`{ [accountName][cusip][year] = qty }`) is required by the UI to compute per-account Before quantities.
- **Stale data prevention**: On new file import, `simpleEl.innerHTML` is cleared before rendering fresh Layer 1 output, ensuring no prior run's rows persist.
- **Unified table UI**: `renderMultiAccountOutput()` injects `tr.acct-subrow` elements after each `tr[data-ri]` main row in `#simple-table`. Each sub-row spans all columns and shows `AccountName [T/R/X]  Before: N  After: N  BUY +N / SELL -N / HOLD` on a color-coded background. Main row background reflects the dominant-allocation account.
- **Collapse/expand**: Clicking the CUSIP cell (`td:first-child`) of a main row toggles its sub-rows (▼/▶ indicator). "Expand All" / "Collapse All" buttons in `#multi-account-panel` operate globally.
- **Account metadata input**: Account name, type, and size are extracted from broker CSV on import. Type is auto-detected from name keywords ("ROTH" → Roth IRA, "IRA" → Traditional IRA, else Taxable). Size defaults to TIPS holdings value; slider max is total account value.

## Example Walkthrough

**Setup:**
- Ladder target (Layer 1): 10 TIPS × 2027, 10 TIPS × 2037, 10 TIPS × 2052 (total 30 TIPS)
- Taxable account: $50k (capacity ~25 TIPS)
- Traditional IRA: $200k (capacity ~100 TIPS)
- Roth IRA: $100k (capacity ~50 TIPS)

**Tier Assignment:**
- shortTier: {2027}
- mediumTier: {2037}
- longTier: {2052}

**Step 5.1-5.3 Allocation:**
- 2027 (short): 10 TIPS → Taxable (preferred for short) ✓
- 2037 (medium): 10 TIPS → Roth (preferred for medium) ✓
- 2052 (long): 10 TIPS → Traditional (preferred for long) ✓

**Step 5.5 Cash Flows (assumed current holdings are 0):**
- Taxable: Buy 10 × 2027 = $10k need; Sell $0 → netCash = −$10k ✗
- Roth: Buy 10 × 2037 = $10k need; Sell $0 → netCash = −$10k ✗
- Traditional: Buy 10 × 2052 = $10k need; Sell $0 → netCash = −$10k ✗

**Result**: Infeasible. No account has proceeds to fund purchases (all are buys). User must either:
- Deposit cash into accounts
- Reduce DARA
- Adjust ladder range to match account values
