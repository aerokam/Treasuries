# 4.0 Computation Modules

## Dependencies
**Requires:** TIPS Basics, 2.0 TIPS Ladders, 3.0 TIPS Ladder Rebalancing

**This file specifies the module interfaces for shared computation libraries.**
The underlying math is defined in the files above; this file defines function contracts,
input/output shapes, and which spec section each function implements.

---

## Purpose

Shared modules eliminate duplicated formula code across `rebalance-lib.js` and `build-lib.js`.
Each function has a single authoritative implementation. Both orchestrators import from these modules.

Traceability: every function maps to a spec formula. If a computed value is wrong, check the
spec formula first, then the function implementation.

## bond-math.js

Pure per-bond calculations. No ladder state, no running pools. Given a bond and refCPI, returns
all derived per-bond quantities. Formulas defined in **2.1 TIPS Basics**.

### `bondCalcs(bond, refCPI)`

**Inputs:**
- `bond` — `{ price, coupon, baseCpi, maturity: Date }`
- `refCPI` — number (settlement date reference CPI)

**Returns:**
```
{
  indexRatio,        // refCPI / bond.baseCpi                        → 2.1 Index Ratio
  principalPerBond,  // 1,000 × indexRatio                           → 2.1 Inflation-Adjusted Principal
  costPerBond,       // price/100 × indexRatio × 1,000               → 2.1 Cost per Bond
  nPeriods,          // 1 if maturity month < 7, else 2              → 2.1 Last-Year Interest (TIPS)
  couponPerPeriod,   // bond.coupon / 2
  ownRungInt,        // principalPerBond × couponPerPeriod × nPeriods → 2.1 Last-Year Interest (TIPS)
  piPerBond,         // principalPerBond + ownRungInt                 → 2.1 P+I per Bond (TIPS)
  annualInt,         // principalPerBond × bond.coupon               → 2.1 Inflation-Adjusted Annual Interest
}
```

**Note:** `piPerBond` here equals `principalPerBond × (1 + coupon/2 × nPeriods)` — algebraically
identical to `adjustedPrincipal + lastYearInterest`. Both forms are valid; see 4.0 Named Quantities.

---

### `calculateDuration(settlement, maturity, coupon, yld)` → Macaulay Duration

Returns Macaulay duration in **years**, or `null` if inputs are degenerate.

**Convention:** Actual/Actual day count, semi-annual compounding (frequency = 2). Matches Google Sheets `DURATION(settlement, maturity, coupon, yld, 2, 1)`.

**Guard conditions:**
- Returns `null` if `settlement >= maturity`
- Returns `null` if `yld ≤ −2` (prevents division by zero in `1 + yld/2`; negative yields between −2 and 0 are **valid** — not filtered out)

**Algorithm:**

1. Find `nextCpn` = first coupon date on or after `settlement` (semi-annual schedule aligned to maturity month and day).
2. Find `lastCpn` = `nextCpn` minus one semi-annual period.
3. Compute period variables:
   - `E   = days(lastCpn, nextCpn)`    — length of current coupon period in days
   - `DSC = days(settlement, nextCpn)` — days from settlement to next coupon
   - `w   = DSC / E`                   — fractional first period (0 < w ≤ 1)
4. Enumerate coupon dates `nextCpn, nextCpn+1period, …` up to and including `maturity`. Let `N` = count.
5. For each period index `j = 0, 1, …, N−1`:
   - `t_j  = w + j`                              — time to payment, in semi-annual periods
   - `CF_j = coupon/2 × 1000`                   — semi-annual cash flow (per $1,000 face)
   - Final period adds principal: `CF_{N-1} = coupon/2 × 1000 + 1000`
   - `PV_j = CF_j / (1 + yld/2)^{t_j}`
6. Return `Σ(t_j × PV_j) / Σ(PV_j) / 2`

The final `/ 2` converts from semi-annual periods to years.

**Used in:** `calcGapParams` (synthetic gap durations), bracket weight calculations (Phase 3b).

---

### `calculateMDuration(settlement, maturity, coupon, yld)` → Modified Duration

```
mac = calculateDuration(settlement, maturity, coupon, yld)
return mac != null ? mac / (1 + yld/2) : null
```

**Spec reference:** 2.1 TIPS Basics §Duration (definition) and §Yield Calculation Conventions (Actual/Actual, semi-annual). Used in: 4.0 Phase 2 (synthetic gap durations), Phase 3b (bracket durations), WAD.

---

### `rungAmount(qty, piPerBond, laterMatInt)`

```
rungAmount = qty × piPerBond + laterMatInt
```

**Spec reference:** 4.0 Phase 5 ARA After formula. Also the structure of `fy_amt` in build-lib.
Ensures the `laterMatInt` term is never omitted in amount calculations.

## gap-math.js

Gap year analysis and bracket sizing. Formulas defined in **4.0 Phase 2 and Phase 3**.

### `interpolateYield(anchorBefore, anchorAfter, targetDate)`

Linear yield interpolation between two anchor bonds.

```
yield = anchorBefore.yield +
        (targetDate - anchorBefore.maturity) × (anchorAfter.yield - anchorBefore.yield) /
        (anchorAfter.maturity - anchorBefore.maturity)
```

**Spec reference:** 4.0 Phase 2, 3.0 Synthetic TIPS Construction.

---

### `syntheticCoupon(yield)`

```
syntheticCoupon = MAX(0.00125, FLOOR(yield × 100 / 0.125) × 0.00125)
```

**Spec reference:** 4.0 Phase 2, 3.0 Synthetic TIPS Construction.

---

### `calcGapParams(gapYears, tipsMap, settlementDate, refCPI, dara, effectivePrelim, pliCreditByGapYear?)`

For each gap year (long→short): interpolate yield → synthetic coupon → synthetic maturity (Feb 15) →
MDURATION → gap qty estimate.

**`effectivePrelim`** — the prelim map passed to this function must reflect actual purchasing intent:
- When `preLadderInterest` is active, zeroed funded years must have their `annualInterest` set to `0` before being passed in. Zeroed funded years have no actual TIPS purchased, so they generate no coupon income and must not contribute to gap year LMI.
- Non-zeroed funded years pass through unchanged.

**`pliCreditByGapYear`** *(optional)* — map of `{ year → pliCredit }` for remaining PLI pool after funded year zeroing, distributed short→long across gap years per **2.0 §Gap Parameter Calculation PLI gap year pass**. Omit (or pass empty) when `preLadderInterest` is disabled.

**Returns:**
```
{
  avgDuration,  // cost-weighted mean of synthetic durations: Σ(qty·dur)/Σ qty (fallback simple mean if Σ qty=0) → 2.0 Gap Avg Duration
  totalCost,    // sum of (gapQty × costPerBond) across all gap years (synthetic costPerBond = $1,000)
  breakdown,    // per-year: { year, qty, piPerBond, laterMatInt, dur }
}
```

**Spec reference:** 2.0 §Gap Parameter Calculation.

---

### `bracketWeights(lowerDuration, upperDuration, avgGapDuration)`

```
lowerWeight = (upperDuration - avgGapDuration) / (upperDuration - lowerDuration)
upperWeight = 1 - lowerWeight
```

**Returns:** `{ lowerWeight, upperWeight }`

**Spec reference:** 4.0 Phase 3c.

---

### `bracketWeights3(d1, d2, d3, Dg, origExcess$, gapTotalCost)`

Three-bracket weight solver. `d1` = orig lower duration, `d2` = new lower duration, `d3` = upper duration, `Dg` = gap avg duration.

```
lw2b   = (d3 - Dg) / (d3 - d1)                                // 2-bracket optimal lower weight
w1_raw = gapTotalCost > 0 ? origExcess$ / gapTotalCost : 0
if w1_raw >= lw2b:
    return { origLowerWeight: lw2b, newLowerWeight: 0, upperWeight: 1 − lw2b }  // sell orig lower to optimal
w1     = w1_raw                                                // orig lower is short — freeze it
w2_raw = (Dg - d3 + w1×(d3 - d1)) / (d2 - d3)
w2     = max(0, w2_raw)
w3     = 1 - w1 - w2
```

**Returns:** `{ origLowerWeight, newLowerWeight: w2, upperWeight: w3, feasible: w2raw >= 0 }`

**Freeze rule:** The orig lower is frozen (not sold) only when its current excess is *below* the duration-optimal target (`w1_raw < lw2b`). When `w1_raw >= lw2b` the result is identical to 2-bracket — the orig lower may be sold to the optimal quantity and the new lower gets nothing. This ensures 3-bracket and 2-bracket produce the same result when the orig lower already covers its share.

**Infeasibility note:** When `lw2b > 1` (gap avg duration shorter than the orig lower bracket duration), the fallback to standard 2-bracket weights is triggered by the caller.

**Spec reference:** 4.0 §3-Bracket Mode.

---

### `bracketExcessQtys(totalCost, lowerWeight, upperWeight, lowerCostPerBond, upperCostPerBond)`

```
lowerExQty = ROUND(totalCost × lowerWeight / lowerCostPerBond)
upperExQty = ROUND(totalCost × upperWeight / upperCostPerBond)
```

**Returns:** `{ lowerExQty, upperExQty }`

**Spec reference:** 4.0 Phase 3c (`targetExcess$ per bracket = gapTotalCost × bracketWeight`),
4.0 Named Quantities `excessQtyAfter`.

---

### `gapParamsWithUpperFeedback(args)`

Shared wrapper around `gapParamsCore` that credits the **upper-bracket (2040) excess coupon** into
gap sizing via a fixed-point iteration. Called by BOTH adapters — `calcGapParams` (build, ladder-core)
and `calculateGapParameters` (rebalance) — so equal inputs produce equal output and the build↔rebalance
round-trip stays exact.

`args` are the `gapParamsCore` args **plus** `refCPI` and optional `creditUpperExcess` (default `true`).

```
resolve bracket bonds from tipsMap (lower = highest Jan TIPS < minGapYear; upper = nearest Feb TIPS > maxGapYear)
extra ← 0
repeat (≤ 6×):
    lmi ← lmiAboveByYear with lmi[upperYear] += extra
    params ← gapParamsCore({ ...args, lmiAboveByYear: lmi })
    upperWeight ← bracketWeights(lowerDur, upperDur, params.avgDuration).upperWeight
    upperExQty  ← ROUND(params.totalCost × upperWeight / upperCostPerBond)
    newExtra    ← upperExQty × (1000 × indexRatio × upperCoupon)     // annual excess coupon, real $
    if newExtra == extra: break        // fixed point — params already reflects it
    extra ← newExtra
return params
```

**Why:** the upper-bracket excess matures after the gap, so its coupon is paid through the gap years
and — like every other rung's coupon — must size the gap synthetic quantities down. But `upperExQty`
is sized *from* the gap cost, so the dependency is circular; it contracts to a fixed point in ~2 steps
(the term is ~1% of gap cost). Falls back to plain `gapParamsCore` when bracket bonds can't be resolved
or `creditUpperExcess` is false. The lower bracket (2036) matures before the gap and never feeds it.

**Spec reference:** 2.0 §Gap Year Coverage Model (Upper-bracket coupon feedback).

## build-lib.js — Pre-Ladder Interest Option

**Option:** `preLadderInterest` (boolean, default `false`)

When `firstYear > settlementDate.getFullYear()` and the option is enabled, coupon payments
received from all ladder bonds BEFORE the ladder starts is credited against the earliest
funded years, potentially eliminating the need for TIPS in those years.

### Algorithm

**Step 1 — Prelim sweep** (unchanged, long→short): produces `prelim[year].annualInterest`
and `prelim[year].laterMatInt` for all funded years.

**Step 2 — Compute pre-ladder pool**
```
preLadderYears = max(0, firstYear - settlementDate.getFullYear())
totalAnnualInt = Σ prelim[year].annualInterest  (all years in ladder)
preLadderPool  = preLadderYears × totalAnnualInt
```
The pool represents total coupon payments from all ladder bonds received in the calendar years
[settlementYear, firstYear−1]. It is computed from prelim quantities — the approximation is
slightly conservative (pool may be modestly over-estimated if some years are zeroed).

**Step 3 — Single interleaved PLI pass (short→long across ALL year types):**
Funded years and gap years are processed together in maturity order. Gap years consume pool before longer-dated funded years — e.g., 2037–2039 are processed before 2040, so whether 2040 is zeroed depends on what the gap years left behind.
```
remaining = preLadderPool
pliCreditByGapYear = {}
for year in (rangeYears ∪ gapYears) ascending:
  if year is a funded year:
    need = MAX(0, DARA - prelim[year].laterMatInt)
    if remaining >= need:
      zeroedYears.add(year); remaining -= need
    else:
      partialCredit = remaining; remaining = 0; break
  if year is a gap year:
    // Use full prelim LMI for funded years above — synth LMI from longer gap years
    // not yet known; this is an accepted approximation for PLI distribution purposes
    actualTIPSLMI = sum(prelim[y].annualInterestReal for funded year y > year)
    need = MAX(0, DARA - actualTIPSLMI)
    if remaining >= need:
      pliCreditByGapYear[year] = need; remaining -= need
    else:
      pliCreditByGapYear[year] = remaining; remaining = 0; break
```
After this pass, `zeroedFundedYears` is fully determined — including whether bracket years like 2040 are zeroed.

**Step 4 — Build effective prelim for gap calculation:**
Zero out `annualInterestReal` for all `zeroedFundedYears` before passing prelim to `calcGapParams`.
This ensures gap year LMI reflects only coupon income from TIPS that will actually be purchased.

**Step 5 — Final output:** for each funded year:
- `fundedYearQty = 0` if year ∈ zeroedYears
- `fundedYearQty = max(0, ROUND((DARA - laterMatInt - partialCredit) / pi))` for the partial credit year
- Normal `fyQty(DARA, laterMatInt, pi)` for all subsequent years

Bracket `excessQty` is computed normally and applied on top regardless of zeroing —
duration matching still applies to bracket years even if their funded-year qty is zero.

**Summary fields added when `preLadderInterest` enabled:**
- `preLadderYears` — calendar years before ladder start
- `preLadderPool` — total pre-ladder coupon pool ($)
- `zeroedFundedYears` — array of years with fundedYearQty = 0

## build-lib.js / rebalance-lib.js — Future 30Y Upper Cover AMD

When `lastYear > 2056` and a Future 30Y upper cover bond exists (currently the 2052 TIPS), the accrued market discount on the excess holdings is counted as an additional income source. See **2.0 §Future 30Y Upper Cover AMD** for the full definition and rationale.

AMD is **interest on the full held excess position**, modeled **held-to-maturity** (this is **Option C** — even, not depleted by sales). The 2052 is near-zero-coupon, so its yield arrives as accretion; income each year = `future30yUpperExQty × a(Y)`, where `a(Y)` is that year's per-bond accretion increment on the **full, undepleted** excess — exactly like a coupon earned on the whole position. Selling some 2052s only realizes the cash; it does not reduce the income basis. Applies to **excess holdings only** — funded-year 2052s are held to maturity and receive par, so their discount is realized at maturity and already captured in P+I (counting AMD on them would double-count).

**Computed once before the Phase 4 sweep** (after `future30yParams`, the cover durations, and `future30yUpperExQty` are known) by `future30yUpperAmdSchedule` in `gap-math.js`. Builds a `Map<year → amd$>` via the constant-yield method (imports `priceFromYield` from bond-math.js):

```
indexRatio_settle = refCPI / future30yUpperCoverBond.baseCpi
costPerBond_2052  = future30yUpperCoverBond.price / 100 × indexRatio_settle × 1000
parPerBond        = indexRatio_settle × 1000          // redemption value, settlement-real dollars
matYear           = future30yUpperCoverBond.maturity year

// adjPrice(Y): constant-yield real price at last calendar day of Feb (new Date(Y, 2, 0))
adjPrice(Y) = (date(Y) >= maturity) ? parPerBond
            : priceFromYield(yield_2052, coupon_2052, new Date(Y,2,0), maturity_2052) / 100 × indexRatio_settle × 1000

// Walk EVERY year settlement+1 → matYear: income = full excess × this-year's accretion; basis steps up.
// No roll, no depletion — the held quantity is constant.
prevAdj = costPerBond_2052
for year in (settlementYear+1 .. matYear):
  ap = adjPrice(year)
  amdByYear.set(year, future30yUpperExQty × (ap − prevAdj))     // exQty × a(Y)
  prevAdj = ap
```

**Per funded year Y** (called inside the Phase 4 sweep):

```
function future30yUpperAnnualAmd(Y):        // build
  return amdByYear.get(Y) ?? 0

function future30yUpperAnnualAmdForQty(Y, qty):   // rebalance — scales for before/after holdings
  if future30yUpperExQty <= 0: return 0
  return (amdByYear.get(Y) ?? 0) × qty / future30yUpperExQty
```

The profile is **even, gently back-loaded by convexity**: `a(Y)` rises modestly toward maturity while the held quantity stays constant, so the annual AMD is roughly level with a gentle upward tilt. **Conserving:** `Σ a(Y) = parPerBond − costPerBond_2052`, so lifetime AMD per bond equals the total market discount exactly.

**fyQty formula with AMD** (replaces plain `fyQty` for years where AMD > 0):

```
fyQty(Y) = MAX(0, ROUND((dara − runningLMI − future30yUpperAnnualAmd(Y)) / piPerBond))
```

**Pre-Ladder Interest integration:** When `preLadderInterest` is enabled, the pool is built from two **named, separately-stored** sources so the credit stays traceable:
```
preLadderCouponPool = preLadderYears × Σ prelim[Y].annualInterest      // coupon interest
preLadderAmdPool    = Σ future30yUpperAnnualAmd(Y)  for Y in [settlementYear, firstYear)
preLadderPool       = preLadderCouponPool + preLadderAmdPool
```
Both `preLadderCouponPool` and `preLadderAmdPool` are exported on the summary (build and rebalance) and consumed by the Level-3 "Pre-ladder pool composition" drill (`buildPreLadderPoolDrill`). The zeroing condition for funded year Y uses the in-ladder AMD only:
```
need = MAX(0, DARA − prelim[Y].laterMatInt − future30yUpperAnnualAmd(Y))
```

**Rebalance:** `amdByYear` is built once from the rebuilt (`after`) `future30yUpperExQty` partition. The before-state sweep uses `future30yUpperAnnualAmdForQty(Y, qtyBefore)` (linear scale by held qty); the after-state Phase 4 sweep uses `future30yUpperAnnualAmdForQty(Y, future30yUpperExQty)` = the full schedule.

**Pre-ladder range guard:** `amdBefore` and `amdAfter` are 0 for years outside `[firstYear, lastYear]`. Pre-ladder AMD (years `[settlementYear, firstYear)`) flows only via PLI pool — it does NOT appear as ARA for those years.

**AMD-driven rebalance years:** Any AMD year Y (i.e. Y in `amdByYear`) within `[firstYear, lastYear]` with `future30yUpperExQty > 0` is added to `rebalYearSet` regardless of method (Gap or Full). This ensures AMD reduces the funded-year qty and triggers actual selling in both modes, preventing `amtAfter > DARA`. The range is the full AMD schedule (every year `settlement+1 → 2052 maturity`), not a hardcoded cap.

**PLI-zeroed year sell-all rule:** When the PLI pool fully covers a funded year's need, the target CUSIP qty is set to 0 AND all older TIPS in that same year (if the portfolio holds multiple TIPS maturing the same year) are also sold to 0. Without this, those older holdings remain in the portfolio and their principal+interest appears as phantom income in that year's "Amount After" column.

**Summary fields:**
- `future30yUpperAnnualAmdByYear` — `Map<year → amd$>` (the `amdByYear` schedule) for display/inspection
- `preLadderCouponPool`, `preLadderAmdPool` — the two named components of `preLadderPool`, for the pre-ladder pool composition drill

(The former `future30yUpperN` and `future30yUpperAmdPerBondPerYear` straight-line fields are removed.)

## ladder-core.js — Shared ladder sizing pipeline

The single authoritative implementation of "size the target ladder from a DARA". **Both**
`build-lib` and `rebalance-lib` call `sizeLadder`; neither reimplements funded sizing, the
pre-ladder-interest (PLI) pool, gap/future-30Y duration matching, or the corrected sweep.

### `selectLadderBonds({ tipsMap, firstYear, lastYear, settlementDate, maturityPref? })`

Picks the canonical bond set purely from `tipsMap`: the funded-year bond per year, the 2040
upper bracket, the nearest pre-gap Jan lower bracket, and the future-30Y cover pair (2056 lower,
2052 upper). Returns `{ yearBondMap, rangeYears, gapYears, future30yYears, future30yLowerYear,
future30yUpperYear, future30yLowerCoverBond, future30yUpperCoverBond }`. This is the single source
for "which bonds the target ladder holds", so build and rebalance size against an identical set.

### `sizeLadder({ dara, daraByYear?, firstYear, lastYear, rangeYears, gapYears, future30yYears, yearBondMap, tipsMap, refCPI, settlementDate, settlementYear, preLadderInterest?, future30y cover bonds/years })`

Pure function: per-year DARA + bonds → target ladder. Runs, in maturity order: the prelim sweep
(PLI bucket + gap-LMI estimate), future-30Y duration match + cover excess, the **AMD schedule**
(held-to-maturity 2052 interest, Option C), the **PLI pool** (zeroing + partial credit), gap
duration match + bracket excess, then the corrected long→short funded sweep.

Returns the target quantities and the params/pool fields callers render, including
`corrFYQty`, `zeroedFundedYears`, `partialCreditYear`, `partialCredit`, `pliCreditByGapYear`,
`pliCreditByFundedYear`, `gapParams`, and the future-30Y/AMD fields.

**Single-source consequence:** a build→rebalance round-trip (build's CUSIP/qty output fed back as
holdings, same DARA/firstYear/lastYear) produces **zero trades** and **zero net cash**. Rebalance
derives per-year DARA from the portfolio, calls `sizeLadder`, then diffs the result against current
holdings into buys/sells. Holdings never enter the sizing — only DARA does. Previously each
orchestrator hand-wrote its own PLI pool; they drifted at the partial-credit boundary year (2040),
which was the build↔rebalance symmetry bug.

**Adapters feed shared cores.** `ladder-core`'s `calcGapParams` (build's "LMI above gap" =
prelim coupon) and `rebalance-lib`'s `calculateGapParameters` (LMI from actual holdings + future
cover) are thin holdings/prelim adapters that both call the shared `gapParamsCore` in `gap-math.js`.
The gap sweep itself is defined once.

## Weighted Modified Duration

Both `build-lib.js` and `rebalance-lib.js` compute `weightedAvgDuration` (the portfolio weighted modified duration) and include it in the returned `summary`.

**Formula:**
```
weightedAvgDuration = Σ(mv_i × mDuration_i) / Σ(mv_i)
```

where:
- `mv_i = totQty_i × costPerBond_i` (build: `fundedYearQty + excessQty`; rebalance: `qtyAfter`, where for bracket rows `qtyAfter = fyQty + excessQtyAfter`)
- `mDuration_i = calculateMDuration(settlementDate, bond.maturity, bond.coupon, bond.yield)` from `shared/src/bond-math.js` — called with the same arguments as all other duration calls; no changes to that function
- Sum runs over all TIPS rows in `details`

**Implementation:** `mDuration` is computed per detail row during the details-building loop (while the bond maturity Date is in scope) and stored on the detail object. `weightedAvgDuration` is derived from the details array after the loop completes.

**Display:** `#wad-display` (label: `Avg Mod Dur`) and `#avg-yld-display` (label: `Avg Yld`) in `#ladder-stats-display` in Row 2 of the top card, shown after run in both modes. Modified duration has no unit suffix (dimensionless); yield is displayed as a percentage (e.g. `2.15%`). See 6.0 UI Layout §Row 2.

**`weightedAvgYield` formula:**
```
weightedAvgYield = Σ(mv_i × yield_i) / Σ(mv_i)
```
Same MV weights as `weightedAvgDuration`. `yield_i` is `d.yield` (real yield from market data, stored on each detail row). Included in `summary` from both `build-lib.js` and `rebalance-lib.js`.
