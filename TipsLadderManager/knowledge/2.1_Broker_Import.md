# 2.1 Broker Import

## Dependencies
**Requires:** TIPS Basics (for CUSIP validation and face value context)
**Feeds into:** 3.0 TIPS Ladder Rebalancing (provides the `holdings` input)

## Purpose
Allows users to upload holdings CSV files instead of manually entering data. Files are auto-detected by format and parsed into a unified `holdings` array consumed by the Rebalance engine.

---

## Supported Formats

### Format 1: Fidelity (`Positions*.csv`)
- **Headers:** Expects columns containing `Account Number`, `Account Name`, `Symbol` (or CUSIP), and `Quantity`.
- **Quantity formatting:** Usually expressed as Face Value (e.g., `10,000`). The parser strips non-numeric characters and divides by 1,000 to convert to Quantity.
- **Account grouping:** Uses `Account Name` (or `Account Number` if name is missing).

### Format 2: Schwab (`*All Accounts*.csv` or single account)
- **Headers:** Expects columns matching `Symbol` and `Qty`.
- **Multi-account structure:** Schwab exports often omit the Account column in the data rows. Instead, they use header blocks: `"[Account Name] ...[Last 4 Digits]"`.
- **Quantity formatting:** Also exports face value. Converted via `Math.round(faceValue / 1000)`.

### Format 3: Simple CSV (`cusip,qty`)
- **Structure:** Optional header row (`cusip,qty`), then one row per CUSIP.
- **Quantity:** Total holdings for that CUSIP. No funded/excess distinction.
- **Result:** `{ cusip, qty }` per row. `excessQty` is not set (treated as unknown).
- **Example:**
  ```
  cusip,qty
  91282CPU9,33
  912810QF8,43
  ```

### Format 4: TipsLadderCom CSV (`cusip,qty,year`)
- **Structure:** No header row. Three fields: CUSIP, quantity, funded year.
- **Bracket TIPS appear as multiple rows** — one row per funded year covered (own maturity year + each gap year whose coverage is allocated to that CUSIP).
- **Funded year qty:** The row where field 3 equals the CUSIP's maturity year in `tipsMap`.
- **Excess qty:** Sum of all other rows for the same CUSIP (gap-year cover allocations).
- **Total qty:** Sum of all rows for that CUSIP.
- **Result:** One `{ cusip, qty: total, excessQty }` per CUSIP.
- **Disambiguation:** Detected when the file has no header and field 3 of the first data row is ≥ 2000.
- **Example:**
  ```
  91282CPU9, 8, 2036    ← funded year (8 for own rung)
  91282CPU9, 6, 2037    ← excess cover for gap year 2037
  91282CPU9, 4, 2038    ← excess cover for gap year 2038
  91282CPU9, 1, 2039    ← excess cover for gap year 2039
  912810QF8, 1, 2037    ← upper bracket excess for 2037
  912810QF8, 3, 2038    ← upper bracket excess for 2038
  912810QF8, 4, 2039    ← upper bracket excess for 2039
  912810QF8, 6, 2040    ← upper bracket funded year (6 for own rung)
  ```

### Format 5: Build Export CSV (`cusip,qty,excess`)
- **Structure:** Header row `cusip,qty,excess`, then one row per CUSIP.
- **`qty` field:** Funded year quantity (0 for PLI-zeroed years or if all holdings are excess).
- **`excess` field:** Excess quantity held for gap/future-30Y coverage (0 for non-bracket TIPS).
- **Total qty:** `qty + excess`.
- **Result:** One `{ cusip, qty: total, excessQty }` per CUSIP.
- **Disambiguation:** Detected by the literal header `cusip,qty,excess` (case-insensitive).
- **Example:**
  ```
  cusip,qty,excess
  91282CML2,0,0
  91282CPU9,0,33
  912810QF8,19,24
  912810QP6,20,0
  ```

---

## Format Detection (Auto)

Detection order:
1. If broker-format keywords are found (`Account Number`, `Positions for`, etc.) → Formats 1 or 2.
2. Else if the first non-empty line matches `cusip,qty,excess` (case-insensitive) → Format 5.
3. Else if the first data row has ≥ 3 comma-separated fields and field 3 is an integer ≥ 2000 → Format 4.
4. Else → Format 3 (simple two-column).

---

## Unified Output: `holdings` Array

All formats produce the same structure passed to the Rebalance engine:

```js
[{ cusip: string, qty: number, excessQty?: number }, ...]
```

- `qty` — total holdings for this CUSIP (funded + excess).
- `excessQty` — excess quantity only (present for Formats 4 and 5; absent for Formats 1–3).

When `excessQty` is present, the engine uses it to initialize `bracketTargetFundedYearQtyBefore[year] = qty − excessQty` for that bracket year, rather than deriving it from DARA math. This preserves the funded/excess split from the prior build across a rebalance cycle.

---

## Process Flow

1. **File Selection (`index.html`)**: When `#holdings-file` changes, the file is read via the browser `File` API.
2. **Auto-Detection**: The first 20 lines are scanned. Broker-specific keywords route to `parseBrokerCSV`. Remaining formats are distinguished by header and field-count rules above.
3. **Extraction & Validation (`src/broker-import.js` for Formats 1–2; inline for Formats 3–5)**:
   - Formats 1–2: Skips non-TIPS assets by validating symbols against the live `tipsMap`.
   - Formats 3–5: Validates CUSIPs against `tipsMap`; skips unrecognized CUSIPs.
   - Format 4: Uses `tipsMap` to determine each CUSIP's maturity year and classify rows as funded vs. excess.
4. **Selection Modal (Formats 1–2 only)**: If > 1 account is found, `#account-overlay` appears.
5. **Handoff**: Holdings are merged (summing quantities of identical CUSIPs) and passed into `runRebalance`.

---

## CusipQty Export (Build / Rebalance)

The "Export CUSIP/Qty" button produces a **Format 5** file from both Build and Rebalance output. This file can be directly re-imported into the Rebalance tab on the next rebalancing cycle.

- **Build mode:** `qty` = `fundedYearQty`, `excess` = `excessQty` from the build details.
- **Rebalance mode:** `qty` = `fundedYearQtyAfter`, `excess` = `excessQtyAfter` from the rebalance details.
- Zero-quantity rows for PLI-zeroed years are included (as `qty=0,excess=0`) so the rebalance engine knows these rungs exist at zero cost.
