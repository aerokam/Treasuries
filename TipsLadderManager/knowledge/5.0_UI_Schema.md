# 5.0 UI Schema

## Dependencies
**Requires:** 3.0 TIPS Ladder Rebalancing, 4.0 Computation Modules

**This file specifies the UI column schema, table rendering logic, and popup routing.**
It is the authoritative reference for `render.js` and `drill.js`.

---

## Purpose

A single `COLS` array drives all table rendering for both Rebalance and Build modes.
Adding, removing, or changing a column requires editing one object in `COLS` only —
headers, cell values, sub-row values, totals, and popup wiring all derive from it.

---

## COLS Schema

Each column is defined by one object:

```js
{
  label,        // string — header text and display label
  key,          // string — colKey used for popup routing and data binding
  fmt,          // 'str' | 'fy' | 'amt' | 'qty' | 'sgn' — formatting type
  rebalOnly,    // bool — column appears in Rebalance mode only (default false)
  buildOnly,    // bool — column appears in Build mode only (default false)
  fyLevel,      // bool — value is year-level, not per-TIPS (default false)
                //   group header: uses first row's value (not sum); null renders blank
                //   child TIPS rows: always blank (value is shown only in group header)
                //   excess sub-rows: unaffected (subValue still renders normally)
  value,        // fn(d, r) → number|string — main row cell value
  subValue,     // fn(d) → number|string | null — Gap sub-row cell value (bracket rows only)
  total,        // bool — include in totals row (default false)
  drill,        // bool — cell is drillable (clickable popup) (default false)
  drillCond,    // fn(v, d) → bool | null — override drill condition (default: v != null && v !== 0)
}
```

`render.js` filters `COLS` by mode before rendering:
```js
const activeCols = COLS.filter(c => mode === 'rebal' ? !c.buildOnly : !c.rebalOnly);
```

## Sub-Row Logic

For bracket and Future 30Y cover rows, a sub-row is rendered showing excess quantities. Sub-rows are labelled **"Gap"** for the 2037–2039 gap brackets and **"Future 30Y"** for the Future 30Y cover pair.

**Row ordering:**
- Gap lower bracket: main row first, Gap sub-row second
- Gap upper bracket: Gap sub-row first, main row second
- Future 30Y upper cover (e.g. 2052): main row first, Future 30Y sub-row second (same as lower bracket)
- Future 30Y lower cover (2056): main row first, Future 30Y sub-row second (same as lower bracket)

Both Future 30Y cover TIPS follow the lower-bracket pattern: excess sub-row always below the funded year row.

This ordering places gap coverage visually between the gap bracket rows, and Future 30Y coverage below each cover TIPS:
```
[gap lower bracket main]
[gap lower bracket — Gap sub-row]
... [non-bracket years] ...
[gap upper bracket — Gap sub-row]
[gap upper bracket main]
... [non-bracket years] ...
[Future 30Y upper cover main]   ← light blue
[Future 30Y upper cover — Future 30Y sub-row]  ← light yellow, label "Future 30Y"
... [years between Future 30Y upper and 2056] ...
[Future 30Y lower cover (2056) main]   ← light blue
[Future 30Y lower cover — Future 30Y sub-row]  ← light yellow, label "Future 30Y"
```

**Row identification:**
- Gap brackets: `d.excessQtyBefore > 0 || d.excessQtyAfter > 0` (rebalance) or `d.excessQty > 0` (build)
- Gap upper bracket: `d.fundedYear === summary.brackets.upperYear`
- Future 30Y cover TIPS: `d.fundedYear === summary.future30yUpperYear || d.fundedYear === summary.future30yLowerYear` when `summary.lastYear > 2056`

**3-bracket mode:** New lower bracket (`summary.newLowerYear`) also gets a Gap sub-row, inserted between orig lower Gap and the non-bracket rebalance years.

**Color coding:**
- `tr.bracket:not(.excess-subrow)` → light blue — gap bracket main rows and Future 30Y cover main rows
- `tr.excess-subrow` → light yellow — all excess sub-rows (Gap and Future 30Y); label distinguishes them

**Spec reference:** 3.0 §Future Cover, 3.0 §Funded Year vs Excess (Bracket Years).

## Funded Year Group Header Row

`render.js` groups the `details` array by `fundedYear` and emits one `<tr class="fy-group-header">` immediately above each group's TIPS rows.

### Structure

```
[▼/▶] [fundedYear label]  [agg col 2]  [agg col 3]  ...  [agg col N]
       ↑ colspan=2 (CUSIP + Maturity positions)
```

- **Toggle cell** (`colspan="2"`): contains the expand/collapse glyph (`▼` expanded, `▶` collapsed) and the funded year integer. Bracket years append `*` (e.g., `2036*`). Left-aligned.
- **Numeric cells**: one per active numeric column (all columns except CUSIP, Maturity, and Yield). Shows the group aggregate; see §Aggregation rule below. Excess/sub-row values are **excluded** from the header; excess rows remain visible as child rows with their own values.
- Bracket-year header gets class `bracket` in addition to `fy-group-header`.

### Aggregation rule

Group header cell value depends on the column's `fyLevel` flag:

| Column type | `fyLevel` | Group header value |
|---|---|---|
| Year-level (`amtBefore`, `amtAfter`, `amount`) | `true` | `groupRows[0].value` — first row's value; null renders as blank |
| Per-TIPS (`cost*`, `qty*`, `qtyDelta`, `cashDelta`, `cost`, `qty`) | `false` | `sum(col.value(d))` across all rows in group |

Year-level columns hold a single funded-year quantity that is identical across all TIPS in the group (ARA Before/After, funded year amount). Summing them would double-count when two TIPS share a funded year. If the value is `null` (e.g., the year's ARA is not yet computed for near-maturity TIPS), the header cell renders blank — not zero.

The `totalFn` (tfoot totals row) is unchanged — it still includes excess. Only the group header uses this narrower aggregate.

### Child rows

All TIPS rows belonging to the group are emitted after the header with class `fy-child`. Excess sub-rows (`.excess-subrow`) are also `fy-child` rows — they are children of the same group and collapse/expand with it.

**`fyLevel` columns on child rows:** `amtBefore`, `amtAfter`, and `amount` render as blank `<td></td>` on all main TIPS child rows. The value is already shown in the group header. Excess sub-rows are unaffected — their `subValue` (excess dollar amount) still renders normally.

```html
<tr class="fy-group-header [bracket]" data-fy="2036" data-expanded="true">…</tr>
<tr class="fy-child bracket" data-ri="…">…</tr>          <!-- TIPS main row -->
<tr class="fy-child excess-subrow bracket" data-ri="…" data-sub="1">…</tr>  <!-- excess -->
```

### Expand/collapse behaviour

- **Default:** all groups expanded (`data-expanded="true"`).
- **Click on header row:** toggles `data-expanded` between `"true"` and `"false"`. All `.fy-child` rows with matching `data-fy` get `hidden` attribute set/cleared.
- **Expand All / Collapse All buttons:** set all group headers to the target state and show/hide all child rows accordingly.
- Expand/Collapse All buttons are placed in the form-card actions row (see 6.0 §Collapsible Groups).

### CSS classes summary

| Class | Applied to | Meaning |
|---|---|---|
| `fy-group-header` | `<tr>` | Funded year group header row |
| `fy-child` | `<tr>` | TIPS row or excess sub-row inside an expanded group |
| `fy-group-header.bracket` | `<tr>` | Group header for a bracket year |

**Spec reference:** 6.0 §Collapsible Groups for visual styling.

## Popup Routing

`drill.js` exports a single function:

```js
buildDrillHTML(d, colKey, summary)
```

Returns HTML string for the popup content. Routes by `colKey`:

| colKey(s) | Popup content | Spec reference |
|---|---|---|
| `amtBefore`, `amtAfter`, `amount` | bondVars → qty → principal + ownCoupon + laterMatInt [+ preLadderCreditForYear if > 0, Build only] | 4.0 Phase 5 ARA Before/After |
| `costBefore`, `costAfter`, `cost` | price → indexRatio → costPerBond → qty → total cost | 2.1 Cost per Bond |
| `qtyAfter`, `qty` | bondVars → piPerBond → DARA → laterMatInt → fyQty (+ excess chain for brackets) | 4.0 Phase 4 step 2 |
| `cashDelta` | costPerBond → qtyDelta → cashDelta | 4.0 Named Quantities Cash Delta |
| `gapAmtBefore`, `gapAmtAfter` | "see Duration Calcs" reference → weight value → totalCost → excessQty → piPerBond → gapAmt | 4.0 Phase 3c, Phase 2 |
| `gapCostBefore`, `gapCostAfter` | "see Duration Calcs" reference → weight value → totalCost → excessQty → costPerBond → gapCost | 4.0 Phase 3c |
| `future30yAmtBefore`, `future30yAmtAfter`, `future30yAmt` | "see Future 30Y Duration Calcs" reference → weight (or fallback notice) → future30yTotalCost → excessQty → piPerBond → future30yAmt | 3.0 Phase 3.1 |
| `future30yCostBefore`, `future30yCostAfter`, `future30yCost` | "see Future 30Y Duration Calcs" reference → weight (or fallback notice) → future30yTotalCost → excessQty → costPerBond → future30yCost | 3.0 Phase 3.1 |

### Duration Disclosure Standards

Both **Duration Calcs** and **Future 30Y Duration Calcs** popups must follow a consistent, transparent structure:

1.  **Block Average**: Display the Average Duration of the missing block (Gap or Future 30Y) at the top.
2.  **Hypothetical Breakdown**: List every synthetic/hypothetical bond in the block with:
    *   Maturity Year and assumed date (e.g., "2037 (Jan 15)" or "2058 (Feb 15)").
    *   Individual Modified Duration.
3.  **Average Derivation**: Explicitly show the mean calculation (Sum / Count).
4.  **Bracket/Cover Pair**: List the existing TIPS used for coverage and their Modified Durations.
5.  **Weight Solver**: Show the weights derived from the matching formula.
6.  **Fulcrum Visualization**: Display the `renderDurationBeam` visual.

**Note**: Quantity breakdowns for individual hypothetical bonds must NOT be shown in the Future 30Y Duration popup, as they do not represent actual holdings.

## Formatting Types

| fmt | Description | Example |
|---|---|---|
| `str` | Raw string, left-aligned | `912810FR4` |
| `fy` | Funded year integer or 'Gap' | `2036` / `Gap` |
| `amt` | Dollar amount, rounded, thousands-separated | `42,317` |
| `qty` | Integer bond quantity | `12` |
| `sgn` | Signed integer with + prefix | `+3` / `−5` |

## Totals Row

Only columns with `total: true` show values in the totals row. Others show blank.
Total = sum of main row values + sum of sub-row values where applicable.

**Spec reference:** totals are display-only sums, not spec-defined quantities.
